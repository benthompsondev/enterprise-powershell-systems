# Script Index

This page is the quick map for the whole repo.

The scripts are sanitized demos based on systems I built, supported, or reviewed while working in IT. They use fake CSVs, local output folders, and simulated actions so the workflow can be shared publicly. The useful part is the pattern: take a disorganized operational process, validate the input, plan the work, write output another person can review, and keep enough logging to prove what happened.

Each folder link opens the full project story and run instructions. Each script link goes directly to the PowerShell source. The descriptions below explain both what the script does and the problem that made it useful.

For the bigger GitHub map, including Ledger and the rest of my public portfolio, start with the [Full Portfolio Map](https://github.com/benthompsondev/benthompsondev/blob/main/docs/portfolio-map.md).

## Endpoint Migration

These projects were built around endpoint changes that affected real users. The goal was not only to change a browser or replace a computer. It was to preserve the user state people relied on, make the steps simple enough for technicians with different experience levels, and leave IT with tracking and recovery information when something did not transfer cleanly.

It discovers local user profiles, keeps nested bookmark folders, creates a timestamped HTML backup first, merges instead of replacing, and writes a per-user report and run log. ## Microsoft 365 / On-Prem Email Migration

This toolkit came from a multi-year move from on-prem Exchange to O365 involving thousands of users, hundreds of shared mailboxes, public folders, directory-based licensing, and mailbox issues that appeared during staged migration waves. The scripts are separate because each one solved a different operational problem. Combining those decisions into one generic checker would make the output harder to trust and harder for the migration team to use.

## How To Verify A Script

Each project folder has a `tests/Run-DemoCheck.ps1` file. That is the fastest way to prove the demo still runs.

From the repo root, the full set is listed in the main `README.md`. GitHub Actions runs the same kind of checks automatically.
